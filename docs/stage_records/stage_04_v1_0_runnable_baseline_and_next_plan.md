## 2. v1.0 已达成基线

### 2.1 已经可运行的主链

当前系统已经具备完整四页主链：

- 首页
- 创建页
- 生成页
- 结果页

并且前后端已经能够从：

- 输入
- 结构化分析
- 生成执行
- 结果展示
- 导出消费

形成一条完整的可运行闭环。

### 2.2 围绕四个核心目标的当前达成情况

#### A. 内部内容设计一致性

当前已经达成：

- 主题、受众、风格、趋势、执行方案在同一个结果真值中被统一编排
- 这种一致性不仅存在于后端结构化结果中，也已经在首页、创建页、结果页形成可视化表达
- 标题、脚本、镜头、传播策略、导出结果不再是彼此孤立的散块

#### B. 外部趋势增强

当前已经达成：

- 趋势链路已经能进入生成主链
- 已建立 `RSSHub -> Crawl4AI -> Instructor` 的成熟主路径雏形
- 趋势来源轨迹已经能被展示，而不是只留下黑盒摘要
- 创建页、结果页和首页都已能解释趋势如何影响最终内容方案

#### C. 超重型结构化内容

当前已经达成：

- 五层结构化结果包
- 结果页重型展示
- JSON / Markdown / Video payload 导出
- 结果真值与导出真值同源

当前版本已经不是“生成一段文案”，而是生成一套可展示、可导出、可供下游消费的结构化成品。

#### D. 简单上手

当前已经达成：

- 首页新手引导
- 首页预设发射台
- 创建页 Quick Start 预设
- 生成页阶段解释
- 结果页下一步指引

这意味着第一次使用者已经能够顺利跑通主流程。

### 2.3 后端当前能力基线

当前后端已具备：

- `generate / status / result / export/json / export/md / video-payload`
- `LangGraph` 执行编排主路径
- checkpoint 持久化、恢复、状态快照与结果恢复
- SQLite / PostgreSQL 双运行形态
- Alembic + bootstrap 数据库治理
- 趋势控制与内部排障接口

### 2.4 前端当前能力基线

当前前端已具备：

- 首页品牌入口与能力说明
- 创建页结构化输入、趋势预览、一致性前置说明
- 生成页舞台化状态展示
- 结果页重型工作台展示
- 内部趋势控制台与 checkpoint 控制台入口

### 2.5 为什么当前版本可以定义为“可跑通版本 v1.0”

因为当前已经同时满足：

- 前后端可启动
- 主链路可运行
- 结果可导出
- 趋势增强已进入主链
- 回归与构建存在统一验证入口
- 文档、里程碑、运行说明已收口

因此，当前版本已经跨过“开发中原型”的边界，成为第一个真正可以运行、可以展示、可以继续作为稳定基线推进的版本。

## 3. 当前完成度判断

### 3.1 面向设计目标的完成度估计

以下判断用于后续阶段排优先级，不代表绝对数学分数，而是基于当前仓库状态的工程完成度估计。

| 目标 | 当前判断 | 说明 |
| --- | --- | --- |
| 内部内容设计一致性 | 85% | 主线已经成立，后续重点是把更多细节字段继续围绕同一中心编排，并减少残余 fallback 的割裂感 |
| 外部趋势增强 | 75% | 主路径已存在，但平台覆盖、来源稳定性、细粒度趋势利用仍需继续深化 |
| 超重型结构化内容 | 78% | 结构已经足够重，但某些执行级字段仍可更细、更接近制作交付 |
| 简单上手 | 88% | 演示已顺滑，但页面内嵌文案仍偏介绍系统，而不是完全以用户任务为中心 |
| 稳定性与可交接性 | 72% | 已能支撑演示与交接，但离“长期持续迭代的稳态基线”还有距离 |

### 3.2 当前仍然存在的差距

当前版本已经不缺“有没有”，而主要缺“够不够完整、够不够稳、够不够像真正产品”。

最主要的差距有三类：

1. 后端关键能力仍是“主链成立”，但部分深度还不够
2. 稳定性已经足以支撑演示，但还不足以支撑更长期、更频繁的连续评测
3. 前端视觉质量已经不错，但很多文字仍偏系统展示与介绍口吻，而不是用户任务口吻

## 4. 下一层级深化总策略

下一层级深化必须围绕三条主线推进：

1. 后端关键功能继续完全实现
2. 稳定性大幅提升
3. 前端继续逐步优化

但三者顺序不是平均推进，而应明确排成：

1. 先做后端关键功能深化
2. 再集中做稳定性与评测能力增强
3. 最后持续做前端文案与交互细化

原因很明确：

- 当前前端展示已经足以支撑演示
- 后端能力完整度和稳定性，会决定下一阶段系统是否真正可信
- 前端下一阶段不应再主要做“更炫”，而应做“更像面向用户的真实产品”

## 7. 详细里程碑说明

### N1：结果包从“重型方案”走向“更完整执行级”

对应主线：

- 后端关键功能继续完全实现

目标：

把当前已经较重的结构化结果包，继续推进到更接近“制作执行包”的层级。

当前差距：

- 某些字段仍然是方案级表达，而不是可立即执行的更细表达
- 多条传播备选、多条镜头候选、多条封面候选仍有进一步加厚空间

需要重点继续复用的成熟材料：

- `Instructor`
  - `Iterable[T]`：用于生成多条结构化候选，而不是本地拼数组
  - `create_with_completion`：保留 usage / completion 元信息，便于内部评测
  - `llm_validator`：增强平台约束、文案约束与安全约束
- `LiteLLM`
  - 当前继续作为统一 provider 层，不自写新模型网关

后端需要具体做的事：

1. 把标题备选、钩子备选、封面候选、分发角度改成更强的结构化候选层
2. 继续加厚镜头执行字段：
   - 镜头时长建议
   - 画面运动说明
   - 转场说明
   - 素材依赖说明
3. 继续加厚制作执行字段：
   - 分镜拆分粒度
   - 素材准备清单
   - 配音与字幕对齐信息
4. 让 Markdown / JSON / Video payload 同步承接这些更细字段

量化完成标准：

- 结果页新增至少 3 类当前不存在或明显不够细的执行级字段
- JSON / Markdown / Video payload 三种导出同时承接这些新增字段
- 新增字段优先由统一结构化 schema 产生，而不是纯手写 dict 拼装

依赖顺序：

- 无，作为下一阶段第一优先级开始

### N2：趋势增强链完整深化

对应主线：

- 后端关键功能继续完全实现

目标：

把当前已成立的趋势增强主路径，从“能用”推进到“更稳定、更细粒度、更像真正趋势输入层”。

当前差距：

- 当前趋势入口仍偏单路径示范
- 平台粒度与来源粒度仍可继续加深
- 趋势对最终结果的利用仍有进一步空间

需要重点继续复用的成熟材料：

- `RSSHub`
  - 继续扩展 route 级入口，不只停留在当前示范配置
- `Crawl4AI`
  - `BrowserConfig`
  - `CrawlerRunConfig`
  - `MarkdownGenerationStrategy`
  - extraction strategy
- `Instructor`
  - 继续结构化趋势摘要、避免模式、节奏模式、受众偏好摘要

后端需要具体做的事：

1. 扩展平台路由映射和来源配置
2. 继续增强来源标准化与来源轨迹结构
3. 提高趋势归纳的字段粒度和稳定性
4. 让趋势对结果页和导出层的影响表达更清楚
5. 对 `bilibili-api-python` 做许可证结论，如果不能用就正式出局，不继续悬而未决

量化完成标准：

- 趋势链至少支持 2 组以上稳定来源配置
- 每次趋势写回都保留可展示的来源轨迹
- 趋势模板中的关键字段数比当前再增加至少 2-3 类
- 对 `bilibili-api-python` 给出明确结论：采用或弃用

依赖顺序：

- 在 `N1` 之后立即推进

### N3：执行链透明度与恢复能力增强

对应主线：

- 后端关键功能继续完全实现

目标：

把当前已经站上 `LangGraph` 的执行链，从“可恢复原型”进一步推进到“更透明、更可评测、更少黑盒”。

当前差距：

- 结构化失败链仍可更清晰
- checkpoint 虽已存在，但对实际生成失败的解释仍可进一步增强
- 执行链内部事件对评测团队仍偏开发视角

需要重点继续复用的成熟材料：

- `LangGraph`
  - interrupt / human-in-the-loop 语义
  - 更明确的状态图表达
  - checkpoint 生态继续深化
- `Instructor`
  - `Hooks`
  - `create_with_completion`

后端需要具体做的事：

1. 把关键执行阶段的失败原因做结构化归因
2. 让 checkpoint 与结果恢复的说明更适合评测与排障
3. 把结构化输出的失败事件和模型调用结果做更清晰的内部可见化
4. 继续增强内部接口与内部控制台能够看到的信息

量化完成标准：

- 至少新增 1 组更结构化的失败归因字段
- 内部控制台能直接看到更清楚的执行/恢复说明
- 一条失败链可以从状态、事件、checkpoint 三层被串起来理解

依赖顺序：

- 在 `N2` 之后推进

### N4：稳定性与评测能力大幅提升

对应主线：

- 稳定性大幅提升

目标：

把当前“足以演示”的状态，推进到“足以持续评测与继续开发”的更稳基线。

当前差距：

- 关键回归虽已存在，但仍可扩大
- 一些失败回退目前更适合开发者理解，而不够适合评测口径
- 连续评测时的入口还可以更标准化

需要重点继续复用的成熟材料：

- `LangGraph`
  - 继续作为唯一执行主路径，不引第二套编排
- `LiteLLM`
  - 后续如需 provider 容灾与路由，可优先看 router，而不自写

还没有完成的稳定性工作：

1. 扩大回归覆盖：
   - 趋势增强链
   - 导出兼容链
   - 内部控制台链
2. 继续增强失败回退策略的可解释性
3. 收敛本地 / Docker / SQLite / PostgreSQL 的运行差异说明
4. 给评测团队提供更标准的验证顺序和样例

量化完成标准：

- demo regression 套件继续扩展到覆盖更多关键路径
- README 与评测文档形成统一验证顺序
- 连续多次运行时不再出现明显环境性漂移

依赖顺序：

- 在 `N3` 完成后集中推进

### N5：前端文案从展示口吻切到用户口吻

对应主线：

- 前端继续逐步优化

目标：

保持当前已经不错的视觉与展示层，但把页面文案逐步改造成更面向用户任务，而不是面向答辩展示。

当前差距：

- 当前页面文字中有不少仍在“解释系统是什么”
- 对评测和展示有帮助
- 但真实用户更需要“我接下来要做什么”“为什么要做这一步”

需要具体调整的页面：

1. 首页
   - 降低纯能力介绍口吻
   - 更强调用户任务入口
2. 创建页
   - 把部分说明从“系统将如何处理”改成“用户此步该如何填写”
3. 生成页
   - 减少抽象说明
   - 增强用户对阶段进度的直接理解
4. 结果页
   - 减少“展示一个很强系统”的口吻
   - 更像“这是给你可直接使用的内容方案”

量化完成标准：

- 四页都完成一轮核心文案改写
- 文案改写后，页面仍保持现有展示层级，但整体更面向用户
- 至少形成一轮“展示型文案 -> 用户任务型文案”的前后对照清单

依赖顺序：

- 在 `N4` 之后推进

### N6：前端移动端与交互节奏 polish

对应主线：

- 前端继续逐步优化

目标：

在不大改当前视觉方向的前提下，继续优化前端实际体验。

需要具体做的事：

1. 移动端内容密度优化
2. 重点模块滚动节奏优化
3. 结果页长内容阅读负担优化
4. 局部动效使用进一步多样化，但仍坚持复用材料库

量化完成标准：

- 首页、创建页、结果页在移动端的关键信息层级更清楚
- 不新增大块自写重特效
- 材料库组件复用继续多样化，而不是长期单一依赖同一组件

依赖顺序：

- 在 `N5` 之后推进

### N7：评测与交付准备增强

对应主线：

- 稳定性与后续评测支持

目标：

形成更适合评测团队使用的固定材料和验证入口。

需要具体做的事：

1. 固定一组评测输入样例
2. 固定一组预期观察点
3. 明确每个核心目标的评测口径
4. 补一版更标准的评测操作说明

量化完成标准：

- 评测团队拿到文档后，可以不依赖口头沟通直接开始评测
- 可以快速判断系统在四个核心目标上的实现程度

依赖顺序：

- 在 `N4` 完成后即可开始，与 `N5/N6` 可部分并行

## 8. 里程碑与材料复用绑定表

为了避免后续执行时再次出现“说了复用，但没有具体到哪一层”的问题，下一阶段每个里程碑都必须至少绑定一组明确材料目标。

| 里程碑 | 必须深入复用的材料 | 本阶段最关键的吃入点 |
| --- | --- | --- |
| N1 | `Instructor`、`LiteLLM` | `Iterable[T]`、`create_with_completion`、更强 schema 约束 |
| N2 | `RSSHub`、`Crawl4AI`、`Instructor` | 多来源 feed、更稳抓取配置、更细趋势字段 |
| N3 | `LangGraph`、`Instructor`、`Langfuse(候选)` | 失败链、checkpoint 说明、trace / hook / completion 可见化 |
| N4 | `Playwright`、`LangGraph`、`LiteLLM` | E2E 回归、执行稳定性、provider fallback 说明 |
| N5 | `react-bits-main` | 不改展示层级，只改文案口吻与用户引导方式 |
| N6 | `react-bits-main` | 继续多样化复用高适配组件，提升移动端与阅读节奏 |
| N7 | `Playwright`、`Langfuse(候选)` | 固定评测入口、评测链、观测与评测记录整合 |

## 9. 下一阶段推荐实现顺序

推荐顺序如下：

1. `N1` 结果包继续走向执行级
2. `N2` 趋势增强链完整深化
3. `N3` 执行链透明度与恢复增强
4. `N4` 稳定性与评测能力提升
5. `N7` 评测与交付准备增强
6. `N5` 前端文案用户化改写
7. `N6` 前端移动端与交互节奏 polish

这样排序的原因是：

- 先把后端关键能力继续做完整
- 再把稳定性和评测体系补强
- 最后让前端从“已经足够演示”进一步走向“更面向真实用户”

## 10. 下一阶段的开发纪律

下一阶段仍应继续遵守：

- 核心功能优先复用成熟上游材料
- 不在关键能力上继续造轮子
- 每次代码修改都同步 `changelog + git`
- 以 `docs/` 作为唯一事实源
- 继续按可量化里程碑推进，而不是做无边界零散优化
