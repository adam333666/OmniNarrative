## 2. 当前系统状态总判断

### 2.1 当前系统是什么

当前系统已经不是假链路 Demo，而是一个具备以下真实能力的一期强化原型：

- 前端四页主链已经成立：首页、输入页、生成页、结果页
- 五步向导式输入已经成立
- 生成状态轮询、结果真值、导出链已经成立
- 五层结构化结果包已经成立
- Markdown / JSON / Video payload 导出已经成立
- 数据库真值层、Alembic、状态事件轨迹已经成立
- 趋势模板数据库真值层与手动刷新入口已经成立
- 结构化输出、执行编排、checkpoint 能力已经开始站在成熟上游仓库之上

### 2.2 当前系统不是什么

当前系统仍然不是以下形态：

- 完整生产系统
- 完整后台管理系统
- 自动调度的实时趋势系统
- 成品视频直接生成系统
- 多用户高并发任务系统

这些边界与 PRD 一致，本期不需要把它误判为“未完成”。

### 2.3 当前阶段总判断

当前系统已经进入“以产品完成与演示完成为优先”的阶段。

后续开发的主优先级应明确为：

1. 重点功能完成度
2. 前端展示与演示效果
3. 关键功能的强胶水去自写化
4. 稳定性、测试与环境治理收口

后两项仍要做，但顺位低于前两项。

#### 4.5.1 `Instructor`

- 当前本地状态：已拉取，commit `41f050c7`
- 当前学习状态：已完成第一轮源码级深读
- 对应核心点：
  - 内部内容设计一致性
  - 超重型结构化内容
  - 外部趋势的结构化归纳
- 当前已复用：
  - `profile_parser`
  - `narrative_generator`
  - `package_assembler`
  - `trend_collector`
  - `structured_output_gateway`
- 当前仍未充分复用的成熟能力：
  - `from_provider` 的统一 provider / model 切换路径
  - `create_with_completion` 的原始 completion 与 usage 追踪
  - `Hooks` 事件系统
  - `Partial` 流式结构化输出
  - `Iterable[T]` 列表式结构化输出
  - lists / enums / TypedDict / multimodal 等 schema 变体
  - retry / reask / semantic validation / hooks
  - citation / raw response / maybe wrapper 等结构化辅助能力
- 后续复用决策：
  - 继续把结果包加厚、趋势归纳加厚时，优先沿 `Instructor + schema` 深挖
  - 不再新增第二套仓内 JSON 解析或大模板系统
  - 若后续需要“边生成边展示”的更重结构化产物，优先评估 `Partial` 流式输出
  - 若后续需要“单次生成多条传播角度 / 多条镜头建议 / 多条封面候选”，优先评估 `Iterable[T]` 而不是仓内手写列表拼装
  - 若后续需要更可观测的结构化失败链，优先接入 `Hooks`，而不是再扩写仓内埋点分支
  - 若后续需要在结果页或内部页展示 token / usage / 原始 completion，优先使用 `create_with_completion`
- 本轮源码级结论：
  - `Instructor` 的真实主干是 `core.client + core.retry + processing.response + dsl.partial/iterable + hooks + validation` 的组合，不是一个薄封装函数
  - `core.retry` 已经把 `ValidationError / JSONDecodeError / InstructorValidationError` 统一纳入 reask 主路径，因此后续继续加厚结构化结果时，不应回退到仓内手写 retry
  - `dsl.partial` 适合未来“生成页或结果页边生成边显示结构化草稿”的需求，但其流式 partial 阶段不支持普通 validators，设计时要接受“完成后再做完整校验”
  - `dsl.iterable` 适合未来把传播策略、标题候选、镜头候选做成真正的流式或批式结构化列表，而不是本仓手写数组拼装
  - `Hooks` 可用于 completion 参数、原始响应、parse error、最后一次失败等事件观测，适合后续把结构化输出调试能力更优雅地接进内部页
  - `llm_validator` 与 Pydantic validator 可直接组合，后续若出现“标题安全、封面文案约束、平台禁忌词约束”之类需求，优先沿这一层做，而不是造第二套规则机
- 计划挂靠里程碑：
  - `M-P0-A`
  - `M-P0-C`
  - `M-P1-B`
  - `M-P1-C`

  - LangGraph 已经内建 interrupt / human-in-the-loop / memory / subgraph persistence 的延展空间，后续若继续增强内部可视化，应沿这条线深化，不再另造状态机

## 11. 下一步直接执行建议

当前里程碑主链已经全部收口。

如果从当前状态继续开发，后续不再是“补齐本期核心能力”，而是进入以下两类工作：

1. 基于已完成系统做演示 polish 与移动端细化
2. 进入下一阶段的新能力规划，但必须重新立项，不应继续沿用当前收口型里程碑

### 11.1 里程碑与上游仓库绑定规则

后续继续开发时，不只要报“在做哪个里程碑”，还要同时报“当前主要依托哪个上游仓库”：

- `M-P0-A / M-P0-C`：优先依托 `Instructor + LiteLLM`
- `M-P1-A / M-P1-C`：优先依托 `RSSHub + Crawl4AI + Instructor`
- `M-P1-B`：优先依托 `Instructor` 的趋势结构化真值与既有前端胶水
- `M-P4-A / M-P4-B`：优先依托 `LangGraph`
- `M-P5-A`：优先依托 `LangGraph + LiteLLM` 已有主路径，不新增第二套底层

### 11.2 后续继续开发前的上游学习顺序

为避免继续出现“只吃到成熟库表层能力”的问题，后续开发前应按下列顺序完成源码级学习与复用决策补完：

1. `Instructor`
2. `LangGraph`
3. `PydanticAI`
4. `RSSHub`
5. `Crawl4AI`
6. `LiteLLM`
7. `bilibili-api-python`

要求：

- 每完成一个库的深读，都要把“当前已用能力 / 未用成熟能力 / 后续接法”回写到本文档
- 若某里程碑准备引入新的上游能力，必须先在本文档对应仓库条目中落明，再开始代码实现

如果用一句话概括当前最合理的推进路径：

- 先补齐结果页与趋势增强这两条最能体现产品价值的主卖点
- 再补首页和首次上手体验
- 最后集中做内部工具统一与稳定性交接
